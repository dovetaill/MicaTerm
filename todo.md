# Snippets Panel TODO

## 当前已确认方案

- 整体视觉方向: `B` `Command Console Fluent`
  - 整体骨架采用 `Mica Alt` 倾向的工具型 Fluent 方案
  - 终端工作区保持克制, 不做过重卡片化
  - 右侧效率面板与顶部命令区承担主要视觉精致度
- 主题策略: `C` `Dark-Primary, Light-Compatible`
  - Slint 样式使用 `fluent`, 不锁定 `fluent-dark`
  - 自定义 design tokens 从一开始定义 `dark/light` 两套
  - 验收优先级为 `dark` 主验收, `light` 保证完整、清晰、不过曝
- 品牌表达策略: `B` `Balanced Brand Guardrail`
  - 第一品牌触点: `Active tab`、顶部命令区入口、右侧 panel 的 segmented control / active state
  - 第二品牌触点: `Activity Bar` 当前选中项、空状态插画 / 辅助图形
  - 刻意保持中性: 终端网格区、host 列表正文、普通文本、focus / danger / warning 状态
  - 首版无障碍底线: 主要操作可键盘到达, `dark/light` 下 hover/active/focus 清晰可辨, 高对比模式不崩、不失读
- 首版微亮点策略: `B` `Curated Highlights`
  - 首版重点打磨: 顶部命令区入口、`Active tab`、右侧 panel `segmented control`、`Welcome / empty state`、`Command Palette`
  - 首版只做功能性完成: host 列表正文、普通表单、普通按钮、普通状态文本
  - 首版明确不做: 大面积背景纹理、强动态插画、复杂 page transition、终端区任何装饰性品牌表达
- `5C` 数据组织: 单层 `group` + 多个 `tags`
- `6B` 搜索与排序: 轻量 fuzzy search, 排序权重为 `pinned > exact/prefix > fuzzy > recent-used`
- `7B` 模板变量: 简单占位符模板, 例如 `{{host}}` `{{user}}` `{{port}}` `{{cwd}}`
- `8B` 作用域: 全局片段 + 主机级 / Profile 级绑定
- `9A` 文件格式: 单文件 `snippets.toml`
- `10B` 编辑弹窗: 平衡表单, 包含 `title` `description` `group` `tags` `scope` `body` `pinned` 与变量识别
- `11B` Insert / Run 交互: 列表单击仅选中, `Enter` 或双击 `Insert`, 预览区按钮执行 `Insert` / `Run`, `Ctrl+Enter` 触发 `Run`

## 后续 TODO 路线图

- 整体视觉方向: `C` `Premium Card Studio`
  - 目标: 在保持核心工作流不变的前提下, 增强品牌化表达、卡片层次、浮层质感与展示性
  - 使用建议: 不作为首版默认方向, 仅作为未来视觉升级或品牌化分支探索
  - 注意点: 需要严格控制卡片密度与阴影强度, 避免喧宾夺主影响终端主工作区
- 长期保持克制 / 中性的区域
  - 终端正文区: 持续保持中性, 不承担品牌化表达
  - host 列表正文: 以信息可读性和扫描效率优先, 不做强品牌化
  - 普通文本: 保持系统级对比关系, 不使用过强 accent 染色
  - focus / danger / warning 状态: 优先系统一致性与可访问性, 不承载品牌表达
- 高对比主题完整适配
  - 目标: 针对 Windows 高对比/可访问性场景提供完整的颜色、焦点、边框与状态样式适配
  - 使用建议: 首版先确保不崩、不失读, 完整高对比体系放入后续专题设计与验证
  - 注意点: 需要单独验证 focus ring、选中态、tab active、drawer/popup 边界和文本可读性
- `6C` 全文索引式搜索
  - 目标: 当片段数量明显增长后, 支持标题、标签、正文、注释的统一检索
  - 触发时机: 当前 `6B` 的命中率、排序质量或大规模片段库体验开始下降时
- `7C` 完整模板引擎
  - 目标: 支持条件分支、循环、复杂变量拼接与更强的可复用命令模板
  - 触发时机: 简单占位符模板无法覆盖批量运维、复杂部署、参数化脚本场景时
- `8C` 全局 + 主机 + 远端路径/OS/标签自动感知
  - 目标: 根据连接上下文自动筛选和推荐最相关的片段
  - 触发时机: 多主机、多系统、多目录工作流导致片段误用或切换成本升高时
- `9C` 目录式 `snippets/<group>.toml`
  - 目标: 按 `group` 拆分片段文件, 改善大规模片段库下的 diff、冲突处理与局部维护体验
  - 触发时机: 单文件 `snippets.toml` 明显膨胀, 或需要更细粒度导入导出与团队协作时
- `10C` 高级编辑器
  - 目标: 在编辑弹窗中增加别名、图标/颜色、OS 过滤、shell 类型、变量默认值、预运行预览等高级元数据
  - 触发时机: 基础平衡表单无法满足复杂片段建模, 或需要更强的运维模板管理能力时

## 当前方案的优点

- 实现边界清晰, 适合空仓库阶段先完成核心链路
- 依赖更少, 更容易和 Slint + Tokio + terminal session actor 骨架对齐
- UI 反馈更稳定, 便于先验证 Drawer 交互、搜索体验和 Insert/Run 流程
- 配置文件可读可改, 方便后续手工迁移和调试
- 复杂度可控, 不会过早引入索引系统、模板 DSL 或自动感知状态机
- 主题策略允许首版优先打磨 dark 模式, 同时避免未来补 light 模式时大面积返工
- 单文件 `snippets.toml` 更适合首版快速验证 schema 和读写链路
- 平衡表单足够支撑当前 `group + tags + scope + variables` 模型, 不会把编辑体验做得过重
- `11B` 交互能兼顾效率与误触控制, 与终端类产品的工作流更匹配

## 后续 TODO 方案的优点

- `Premium Card Studio` 能增强品牌区分度, 更适合后续做视觉升级或展示型工作台体验
- 完整高对比适配能提升无障碍质量与系统一致性, 也能减少极端主题下的可读性问题
- `6C` 能显著提升大规模片段库下的检索能力和可发现性
- `7C` 能提升高级用户的模板复用能力, 减少重复维护相似命令
- `8C` 能让片段更贴合当前连接上下文, 降低误发命令和手动筛选成本
- `9C` 能降低大型片段库的文件冲突概率, 也更利于按组迁移和模块化管理
- `10C` 能让 Snippets 从“轻量命令库”进一步进化为“结构化操作资产编辑器”
- 三者组合后, Snippets 会从“命令收藏夹”升级为“上下文感知的操作面板”

## 为什么当前不直接上 TODO 方案

- `6C` 需要额外的索引构建、增量更新和排序校准, 首版投入偏高
- `7C` 会引入模板语义边界、调试成本和错误提示设计, 首版很容易做重
- `8C` 依赖终端会话、主机元数据、远端环境探测与缓存策略, 需要更完整的应用骨架
- `9C` 会增加文件发现、分组合并、重命名迁移与原子写入策略的实现成本
- `10C` 会抬高编辑器状态复杂度, 包括字段校验、条件显示、预览一致性与 UX 负担
- `Premium Card Studio` 更依赖精细的层级、阴影、留白与动效调校, 首版容易稀释工具效率导向
- 完整高对比适配需要额外维护一套状态映射与验证矩阵, 首版投入不适合和核心骨架并行推进
- 如果在首版同时引入这三项, 很容易稀释对核心体验的验证, 包括: 打开速度、插入路径、运行链路、Drawer 可用性
